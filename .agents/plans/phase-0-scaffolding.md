# Feature: Phase 0 — Repo Scaffolding & Hello-World End-to-End

The following plan should be complete, but its important that you validate documentation and codebase patterns and task sanity before you start implementing.

Pay special attention to naming of existing utils types and models. Import from the right files etc.

## Feature Description

Stand up the polyglot monorepo skeleton for Interior Flow 3D so that every layer (Next.js frontend, FastAPI backend, Supabase, Azure infra, GitHub Actions CI/CD) boots end-to-end with no business logic. After Phase 0, a developer can run `pnpm dev` + `uv run uvicorn app.main:app --reload` locally and hit a deployed `/healthz` from the browser. No catalog, no LLM, no slot logic — those live in Phases 1–3.

This is the "stack honesty" moment from PRD §2: prove the boring productive stack works end-to-end before any AI work. Goal of phase: dev loop fast, CI/CD green, Azure resources exist, Supabase project + initial migration applied.

## User Story

As the developer (Lucian)
I want a fully-scaffolded monorepo with a hello-world running on every layer
So that subsequent phases (slot resolver, LLM, auth) implement against working infra instead of also fighting setup

## Problem Statement

Repo currently has only `.claude/` config + `CLAUDE.md`. No `frontend/`, `backend/`, `supabase/`, `infra/`. No CI. No deployments. Cannot start Phase 1 (slot resolver, R3F viewer) until project structure, build tooling, deploy targets, and dev commands all exist and are validated.

## Solution Statement

Create the four top-level directories per the locked structure in `CLAUDE.md` + `.claude/PRD.md` §6, fill each with the minimum needed to boot, wire two CI jobs, deploy once to confirm, commit. No production logic — every endpoint returns a stub, every page is a placeholder, but every tool in the stack is exercised.

## Feature Metadata

**Feature Type**: New Capability (greenfield scaffolding)
**Estimated Complexity**: Medium (breadth high, per-layer depth low)
**Primary Systems Affected**: All — `frontend/`, `backend/`, `supabase/`, `infra/`, `.github/`, repo root
**Dependencies**: pnpm 9, Node 20, Python 3.12, uv, Azure CLI, Supabase CLI, Docker, GitHub Actions

### Patterns to Follow

**Naming Conventions (per CLAUDE.md):**
- Python: `snake_case` for files/funcs/vars; `PascalCase` for Pydantic models + classes.
- TypeScript: `kebab-case` files; `PascalCase` React components + types; `camelCase` vars/funcs.
- Env vars: `SCREAMING_SNAKE_CASE`. Frontend public vars prefixed `NEXT_PUBLIC_`.
- Style enum values: lowercase. Preference enum values: `snake_case`. Slot ids: `snake_case` (walls/floor) + `corner_NE`-style uppercase compass for corners.

**Backend factory (`api.md` §10):**
```python
def create_app() -> FastAPI:
    settings = get_settings()
    app = FastAPI(title="Interior Flow 3D API", version="0.1.0")
    app.add_middleware(
        CORSMiddleware,
        allow_origins=settings.CORS_ORIGINS,
        allow_credentials=True,
        allow_methods=["GET", "POST", "DELETE", "OPTIONS"],
        allow_headers=["authorization", "content-type"],
        max_age=600,
    )

    @app.get("/healthz")
    async def healthz() -> dict[str, str]:
        return {"status": "ok"}

    return app

app = create_app()
```

```python
class Settings(BaseSettings):
```

Note: empty defaults so app boots locally without secrets in Phase 0. Phases 2–3 enforce `min_length=1`.

**Frontend `authedFetch` (`components.md` §5):**
```ts
const API = process.env.NEXT_PUBLIC_API_BASE_URL!;

export class ApiError extends Error {
  constructor(public status: number, message: string) { super(message); }
}

export async function authedFetch<T>(path: string, init?: RequestInit): Promise<T> {
  const res = await fetch(`${API}${path}`, {
    ...init,
    headers: { "content-type": "application/json", ...init?.headers },
  });
  if (!res.ok) throw new ApiError(res.status, await res.text());
  return res.json();
}

export const fetchHealth = () => authedFetch<{ status: string }>("/healthz");
```

**Layering rules (api.md §1):**
- `routers/` may import from `models/`, `services/`, `deps.py`. Nothing else.
- `services/` may import from `models/` and other `services/`. Never from `routers/` or `fastapi`.
- `models/` only `pydantic` + `typing`. No I/O, no business logic.

## NOTES

**Why empty-string defaults in `Settings`:** Phase 0 must boot without any Azure / Supabase secret. Phases 2 (LLM) and 3 (auth) tighten via `min_length=1` validators on the relevant fields when those layers come online. Treating Phase 0 as "no secrets needed" keeps the dev loop instant.

**Why placeholder image in Bicep:** Container App resource cannot deploy without an image reference. The public `azuredocs/containerapps-helloworld` lets infra deploy first; the `backend` workflow's first main-branch run pushes the real image and updates the revision. Avoids chicken-and-egg with ACR.

**Why no ACR module in Bicep:** Phase 0 keeps infra minimal — assume ACR + RG pre-exist (or are created via `az acr create` once, out of band) and only manage the per-environment resources (KV, ACA env, ACA app) in Bicep. Documented in README.

**Why frontend `app/app/page.tsx` is a Client Component despite components.md §1 pushing client to leaves:** the placeholder calls TanStack Query at the page level for Phase 0 only. Phase 2 splits the wizard into per-step Client Components and keeps the page itself a Server Component. Note the deviation in the Phase 2 plan.

**Why `supabase init` first vs hand-writing `config.toml`:** running `supabase init` is the supported path; the resulting `config.toml` is checked in. Manual edits afterward (e.g. tweaking ports) are fine.

**Trade-offs accepted:** No Storybook, no Playwright, no axe-core, no OTEL, no rate limit, no observability stack, no shadcn install (Phase 2 brings it). Per PRD §4 "Out of Scope — Technical".

**Confidence Score: 8/10.** Risks: (1) Bicep syntax drift between azurecli versions; (2) `amondnet/vercel-action` may need replacement if Vercel deprecates again — verify before running; (3) Supabase CLI requires Docker Desktop on Windows; document in README. None block one-pass success if the validation commands are run top to bottom.
